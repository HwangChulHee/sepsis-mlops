## HOLD (수정 필요)

### HOLD-1 — 통계 프레임이 자기상관·도구(Evidently)와 불일치 (★거짓경보 정확성)

- **항목**: 결정 3(KS), 결정 4(α=5%·다중검정 보정), 결정 7(Evidently), PASS #3·#6
- **문제**:
  1. **시점 자기상관 → iid 가정 붕괴.** 드리프트 관측 단위가 **per-timestep raw 입력**이다(`serve/metrics.py:30-40` `record`는 매 `/predict`마다 시점값 관측). 같은 환자의 연속 시간(ffill carry 포함)은 **강한 자기상관** → KS p값·α=5% 통제가 가정하는 **독립표본이 아니다**. 따라서 결정 4의 "다중검정 보정으로 거짓양성률 5% 고정"은 **성립하지 않는다**(유효표본수 과대 → 실제 FPR ≫ α). DDD는 검정 *단위*(시점 vs 환자)를 결정하지 않았다.
  2. **Evidently는 대표본에서 KS를 안 쓰고, α 보정을 제공하지 않는다** [확인됨: §1차]. 수치형 >1000 obs 기본은 **Wasserstein**(거리), ≤1000만 KS, 저cardinality(Gender)는 JS. 임계는 거리 임계(기본 0.1), 데이터셋 드리프트는 **"드리프트된 컬럼 비율" 규칙** — **p값·α·Bonferroni 개념이 없다**. 즉 결정 3(KS)·4(α 보정)는 결정 7(Evidently)의 거리기반 설계와 **충돌**. KS를 강제 오버라이드하면 대표본 over-power가 되살아나고(결정 3이 우려한 바로 그것), 그걸 막으려 Evidently는 *원래* Wasserstein을 쓴다.
  - 종합: "α=5% + 다중검정 보정"은 (a) 자기상관으로 분석적으로 무효, (b) 선택한 도구가 제공하지도 않음 → **이중으로 깨진 프레임**.
- **근거**: `h4_drift_decisions:45,54-59,84`; `serve/metrics.py:30-40`(per-timestep 관측); §1차(Evidently 기본 Wasserstein/거리·α 보정 없음).
- **제안**:
  - **거리기반으로 전환**(Evidently 정렬): 수치형 = **Wasserstein**(또는 PSI), 저cardinality/Gender = JS/PSI. 거리는 **효과크기 내재**라 대표본 over-power가 없음(결정 3의 KS 효과크기 하한이 풀려는 문제를 도구가 구조적으로 해결).
  - **거짓경보 통제는 분석적 α가 아니라 *경험적 보정*으로**: 임계값을 **합성 드리프트 주입(PASS #6)**으로 보정해 목표 거짓경보율을 맞춘다. "α=5%"는 *경험적 거짓경보 예산*으로 재정의(분석적 per-test α 아님). 데이터셋 알람은 Evidently의 "드리프트 컬럼 비율" 규칙 사용.
  - **검정 단위 명문화**: 자기상관 완화 위해 **환자당 1관측(예: 환자별 요약/마지막) 또는 블록/유효표본 보정**. 그래야 PASS #6의 FPR 측정이 의미를 가짐.
  - KS+Bonferroni를 굳이 유지하려면 **Evidently 밖에서 직접 구현 + 유효표본 보정**해야 함(결정 7 "직접 구현 불필요"와 모순 — 명시).

### HOLD-2 — 우산장수(피드백 루프) 라벨 변질에 표기·보정 기법 전무 (★라벨 지연)

- **항목**: 결정 5(결합 알람의 성능 측정), PASS #4
- **문제**: 결합 알람은 "지연 라벨 백필로 측정한 성능 하락"에 의존한다(`:65`). 그런데 **모델이 맞혀 의사가 개입 → 환자 회복 → 라벨이 음성**이 되면, *정확히 모델이 성공했기 때문에* 그 케이스가 **거짓양성(FP)으로 집계**돼 성능이 나빠 보인다(`:67` 스스로 인지). 이 변질은 결합-알람의 성능신호를 **체계적으로 하향 편향** → "성능 하락"이 모델 실패가 아니라 모델 성공일 수 있다. DDD는 "**갖추되 인지**"만 할 뿐 **표기·보정 기법이 없다**(검토요청 `:69`이 물었으나 미응답).
- **근거**: `h4_drift_decisions:65,67,69`; H3-c가 보인 "측정/개입 패턴의 사이트 특이성"(reports/h3_results.md §H3-c)과 같은 결의 교란.
- **제안**: 최소 **태깅 기법**이라도 명문화 — 알람 발화 + (가용 시)개입 발생 케이스를 **intervention-confounded로 표시**하고, 윈도우 성능 계산에서 **제외하거나 별도 집계**(naive FP로 안 셈). 개입 데이터가 없으면 그 한계를 명시하고 **개입-비개입 코호트 분리 측정**을 설계에 넣음. 완전 인과보정은 이연 가능하나, **편향을 그대로 둔 채 결합 알람을 PASS시키지 말 것**(PASS #4에 "intervention-confounded 라벨이 성능신호를 오염시키지 않음" 추가).

## 1차 확인 결과

- **Evidently 드리프트 기본 동작** [확인됨: Evidently 문서/가이드 검색]:
  - 수치형 **>1000 obs → Wasserstein Distance**(기본), **≤1000 → 2표본 KS**, 범주형·저unique → Jensen-Shannon. 기본 임계 **0.1**. `DataDriftPreset(num_stattest='ks'|'psi', thresholds=…)`로 오버라이드 가능.
  - 데이터셋 드리프트 = **"드리프트된 컬럼 비율" 규칙**(예: 50%). **다중검정 α 보정(Bonferroni/FDR) 내장 없음**, 거리기반엔 p값 자체가 없음. → 결정 4의 α 프레임을 Evidently가 제공하지 않음(HOLD-1).
- **대표본 KS over-power** [확인됨: 통계 사실 + statstest류]: n↑이면 사소한 차이도 p<0.001 → 효과크기 필요. 단 Wasserstein/PSI 같은 **거리지표는 효과크기 내재**라 이 함정이 없음(Evidently가 대표본에서 거리로 가는 이유).
- **PSI 0.1/0.2** [확인됨: 업계 rule-of-thumb]: 엄밀 임계 아님 → 도메인 보정 필요(DDD도 인지 `:47`). 우리 보정 수단 = 합성 주입(PASS #6).
- **serve/metrics.py** [확인됨: 코드]: INPUT_FEATURE는 **per-timestep raw 관측**(coarse 버킷), INPUT_MISSING은 결측 카운터 → 드리프트 단위가 per-timestep임을 확정(HOLD-1).

## 실행 전 권고 (비차단)

1. **reference = RAW 입력 분포로 명문화(결정 1)** — 서빙은 raw 입력을 받고 `metrics.py`도 raw 관측 → 기준은 **A-train의 피처별 RAW 관측값 분포 + 결측률**이어야 함. 결정 1의 "μ/σ 동결"은 *정규화*용이지 *드리프트 기준*이 아님(혼동 정리). 정규화 후 분포 감시는 불필요·해석 저하.
2. **결측률 드리프트를 1급 신호로** — H3-c가 보였듯 *측정 패턴*이 사이트 특이적. `INPUT_MISSING`(피처별 결측률) 변화를 PSI/거리와 함께 감시(검사 결측률 급변 = covariate 드리프트의 핵심).
3. **PSI 빈 전략 명시** — raw 윈도우 PSI의 빈 경계 = **reference의 분위수(예: 십분위)** 고정(현재 미정). coarse Prometheus 버킷 재사용 금지(결정 6 정합).
4. **조기경보 = watch 프레이밍**(결정 5/한줄요약) — 한 줄 요약의 "성능 떨어지기 전 경보"는 사실상 **watch(라벨無 조기 신호)**이고, 실제 **action 알람은 라벨 지연만큼 늦다**. 온콜이 알람을 조기로 오해하지 않게 명시.
5. **watch→action 승격 기법 구체화** — watch 이벤트에 **윈도우 id 태그** → 백필 라벨 도착 시 그 윈도우의 **동결 τ 기준 utility/PR-AUC** 계산 → 기준(어디 대비? A-val? 운영 baseline?) 이하면 승격. 성능 기준선 출처 명시.
6. **다중검정/도구 정합(거리기반 채택 시)** — Bonferroni vs BH는 거리기반 전환 시 무의미(권장: 컬럼-비율 규칙 + 경험적 임계). p값 경로 유지 시에만 BH-FDR + 유효표본 보정.
7. **PASS #5 프로그래매틱화** — "Grafana 대시보드"는 시각물(비프로그래매틱) → "Evidently 리포트 생성 + Prometheus 드리프트 메트릭 노출 + Grafana provisioning JSON 존재"로 분해. #6(합성 주입)이 진짜 검증이므로 **임계 보정도 #6으로** 수행.
8. **K = 배포 featureset 크기** — 컬럼 수(9 vitals / 18 vitals_labs)는 로드된 번들 featureset에서 파생(서빙 원자성과 정합).
