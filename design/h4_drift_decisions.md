# H4-드리프트 설계결정문서 (DDD) — covariate 드리프트 감시

> **설계 근거**: H4-서빙 위. `serve/metrics.py`의 입력 피처 히스토그램·결측 카운터(covariate 드리프트 토대) + A-train 분포(H1 동결)를 기준으로 감시. 다음(H4-재학습)의 트리거를 만든다.
> **워크플로우·출처등급**: [`WORKFLOW.md`](WORKFLOW.md). 검토(`h4_drift_review.md`) 통과 후 핸드오프로.
> **상태**: 초안 — 레드팀 검토 전.
> **H4 분할**: 서빙(완료) → 드리프트(본 문서) → 재학습. 의존 순서(서빙돼야 감시, 감시돼야 재학습).

## 한 줄 요약

운영 입력이 학습 분포(A-train)에서 **벗어나는지를 라벨 없이 실시간 감시**한다. PSI(실용 임계)+KS(효과크기 하한)로 피처별 드리프트를 재고, **다중검정 보정으로 알림률을 α=5%에 고정**. 단 **드리프트 단독으로 알람하지 않고**(거짓경보 → 온콜 소진), 지연 라벨 백필로 측정한 성능 하락과 **결합**해야 알람. Evidently로 감지, Grafana로 대시보드. **"성능 하락이 모델 탓이 아니라 데이터 변화 탓일 수 있다"를 성능이 떨어지기 전에 경보**해 재학습 타이밍을 잡는다.

## 범위 / 범위 외

| 범위 (H4-드리프트) | 범위 외 |
|---|---|
| covariate 드리프트 감시(입력 분포 변화) | 재학습 실행·번들 교체 (H4-재학습) |
| PSI+KS+효과크기, α=5% 고정(다중검정 보정) | concept drift 본격 대응(라벨 의존) |
| 드리프트+성능 결합 알람 | 피처/결측 ablation 2차 바퀴 |
| Evidently 통합 + Grafana 대시보드 | 실제 병원 라벨 파이프라인 |
| 지연 라벨 백필·사람 검토 샘플링(설계) | |

## 결정 1: 기준 분포(reference) — A-train 동결

- **결정**: 드리프트 기준 = **A-train 입력 분포**. 모델이 실제 학습한 분포이자 H1에서 통계(μ/σ) 동결한 것. 피처별 reference 히스토그램/요약을 산출·동결.
- **근거 + 출처등급**: "모델이 본 세상"에서 벗어나나가 우리 질문 [우리 결정]. 학습 분포 대비가 표준 [확인됨: 업계 관행 — 다수 출처]. cross-site(A→B) 붕괴를 H3에서 봤으므로, 입력이 A에서 벗어나면 그 붕괴 위험 신호 [확인됨: reports/h3_results.md].
- **고려한 대안**: 운영 초기 N일을 기준(배포 후) → 모델 학습분포와 불일치 가능. 기각.
- **검토 요청 항목**: A-train reference가 서빙 전처리 전(raw)인지 후(정규화)인지 — 어느 단계 분포를 감시하나.

## 결정 2: 현재 윈도우(current) — 최근 N, 의료라 느린 주기

- **결정**: 현재 분포 = **최근 윈도우**(최근 N건 또는 최근 기간). 실시간 수집 + **주기적 분석**. 의료는 저변동 도메인이라 분석 주기를 촘촘히 안 함(분/시간 단위 알람 대신 일/주 단위 분석 기본, 설정 가능).
- **근거 + 출처등급**: 단일 시점은 노이즈, 윈도우로 안정화 [확인됨: 업계 관행]. 분석 주기는 도메인 의존 — 의료 등 저변동은 느림 [확인됨: 1차 출처]. 짧은 윈도우=민감(노이즈)/긴 윈도우=둔감(지연) 트레이드오프 [확인됨].
- **고려한 대안**: 매 예측 즉시 비교(노이즈 폭증). 고정 대형 윈도우(느린 변화 못 잡음).
- **미결/옵션**: 윈도우 크기·분석 주기 구체값은 핸드오프(설정). reference 대비 표본 크기 균형.
- **검토 요청 항목**: 윈도우 표본 크기가 검정 신뢰성과 거짓경보(결정 3)에 미치는 영향.

## 결정 3: 검정 — PSI(실용 임계) + KS(효과크기 하한) ★대표본 거짓경보 방지

- **결정**: 피처별 **PSI + KS** 병용. PSI 실용 임계 **< 0.1 안정 / 0.1~0.2 조사 / > 0.2 조치**. **KS는 p값 단독 금지 — 효과크기(KS 통계량) 하한과 함께** 판정. (우리 표본이 크면 KS가 사소한 차이도 "유의"로 잡아 거짓경보.)
- **근거 + 출처등급**:
  - PSI 임계 0.1/0.2 [확인됨: 업계 관행 — 다수 출처]. 단 관행(rule of thumb)이라 도메인 보정 필요(arXiv 사례 PSI 1.4~5.2) [확인됨: 1차 출처].
  - **대표본 KS 함정**: 표본 크면 p<0.001인데 KS 통계량 0.02처럼 통계적 유의·실질 무의미 → 효과크기 병행 필수 [확인됨: 1차 출처 statstest].
- **고려한 대안**: KS p값 단독(대표본에서 상시 알람 → 거짓경보 폭증). PSI 단독(연속분포 민감도 보완 위해 KS 병행).
- **검토 요청 항목**: PSI 버킷팅이 serve/metrics.py의 coarse 히스토그램 버킷과 정합하나, 아니면 raw 윈도우로 재계산하나(결정 6).

## 결정 4: 알림률 α=5% 고정 + 다중검정 보정

- **결정**: 검정 유의수준 **α=5% 고정**. 피처 9/18개를 각각 검정하므로 **다중검정 보정**(Bonferroni 등)으로 전체 거짓양성률을 5%에 묶음.
- **근거 + 출처등급**: 피처 K개 독립 검정 시 우연히 하나는 걸림(K=18이면 보정 없이 ~60% 거짓양성) → 보정 필수 [확인됨: 통계 일반]. α 고정이 거짓경보·온콜 소진 방지 [확인됨: 1차 출처].
- **고려한 대안**: 피처별 독립 α(전체 거짓양성률 폭증). 보정 없음(알람 피로).
- **검토 요청 항목**: 보정 방식(Bonferroni 보수적 vs Benjamini-Hochberg FDR) 선택 근거.

## 결정 5: 알람 조건 — 드리프트 + 성능 결합 (★드리프트 단독 금지)

- **결정**: **드리프트 감지만으로 알람하지 않는다.** **입력 드리프트 + (지연 라벨 백필로 측정한) 성능 하락**이 결합될 때 알람. 라벨 없는 동안은 **드리프트를 "경보(watch)"로만**, 성능 영향 확인 시 "조치(action)"로 승격.
- **근거 + 출처등급**: 평가 영향 없는 드리프트는 거짓경보이고 온콜을 소진 [확인됨: 1차 출처 futureagi]. 라벨 없는 감시 = 프록시 + 사람 검토 샘플링 + 지연 라벨 백필 [확인됨: 1차 출처 runbook].
- **⚠️ 피드백 루프(우산 장수) 주의**: 모델이 맞혀 의사가 개입→환자 회복→라벨이 "음성"으로 변질될 수 있음. 지연 라벨로 성능 측정 시 이 변질을 인지("갖추되 인지") [우리 결정 — 메모리 원칙].
- **고려한 대안**: 드리프트 단독 알람(거짓경보 천지, 온콜 소진). 성능만 감시(라벨 지연으로 늦음 — 드리프트의 조기경보 가치 상실).
- **검토 요청 항목**: 라벨 지연 구조에서 백필·watch→action 승격이 구현 가능한지, 우산장수 변질을 어떻게 표기하나.

## 결정 6: 데이터 소스 — raw 윈도우 보관 (Prometheus 히스토그램은 대시보드용)

- **결정**: 드리프트 검정은 **raw 입력 윈도우**(최근 N건 원값)로 계산. `serve/metrics.py`의 coarse Prometheus 히스토그램(버킷 [0,1,5,…300])은 **대시보드 시각화·운영 모니터링용**이고, 정밀 PSI/KS는 raw 윈도우로(coarse 버킷은 정밀도 부족). 윈도우 저장은 경량(최근 N만).
- **근거 + 출처등급**: Evidently 등은 reference vs current raw를 받아 검정 [확인됨: 도구 관행]. coarse 버킷으로 PSI 계산하면 버킷 경계 왜곡 [우리 판단]. metrics.py 주석도 "H4-drift can refine" 명시 [확인됨: serve/metrics.py].
- **고려한 대안**: Prometheus 히스토그램만으로 PSI(버킷 거칠어 부정확). raw 전량 보관(불필요, 윈도우면 충분).
- **검토 요청 항목**: raw 윈도우 저장이 서빙 상태(per-pid)와 분리되는지, 메모리/디스크 부담.

---

## 결정 7: 도구 — Evidently + Grafana

- **결정**: **Evidently**로 드리프트 감지(PSI·KS 내장, reference vs current 리포트). 결과를 Prometheus로 노출 → **Grafana 대시보드**(피처별 드리프트 점수·시계열·알람). 서빙 관측성 스택(Prometheus) 재사용.
- **근거 + 출처등급**: Evidently가 PSI·KS·리포트 제공(직접 구현 불필요) [확인됨: 도구]. Grafana는 pdm-mlops·서빙에서 사용 [우리 경험: pdm-mlops·H4-서빙].
- **고려한 대안**: 직접 scipy로 KS/PSI(재구현 부담, Evidently가 표준). NannyML 등 대안(범위).
- **미결/옵션**: Evidently 버전·리포트 형식은 핸드오프(1차 확인).
- **검토 요청 항목**: Evidently 통합이 raw 윈도우(결정 6)·α 보정(결정 4)과 정합하나.

## 부속 결정

- **감시 단계**: covariate(입력)만 H4-드리프트 범위. concept drift(관계 변화)는 라벨 의존이라 H4-재학습/이연.
- **출력 드리프트도 보조 감시**: 예측 확률 분포(serve PRED_PROB)도 같이 봄(입력 외 신호).
- **재학습 트리거는 H4-재학습**: 본 단계는 "언제 재학습해야 하나"의 **신호 생성**까지. 실행은 다음.

## H4-드리프트 PASS 기준 (핸드오프에 박을 게이트 — 초안)

1. A-train reference 분포 산출·동결, 피처별.
2. 최근 윈도우 raw 수집 → PSI+KS+효과크기 계산.
3. **α=5% 다중검정 보정** 적용(보정 전후 알람 수 차이 확인).
4. **드리프트 단독 ≠ 알람**: 알람이 드리프트+성능 결합 조건에서만(드리프트만으론 watch).
5. Evidently 리포트 생성 + Grafana 대시보드 노출.
6. **합성 드리프트 주입 검증**: 알려진 분포 이동을 넣으면 감지, 안 넣으면 거짓경보율 ≈ α.
