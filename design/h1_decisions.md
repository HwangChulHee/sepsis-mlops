# H1 설계결정문서 (DDD) — 전처리·피처·분할

> 작성: 채팅 핑퐁 / 검토 예정: `h1_review.md` (Claude Code 레드팀) / 구현: `h1_handoff.md`
> 워크플로우·출처등급 범례: [`WORKFLOW.md`](WORKFLOW.md) 참고.

## 한 줄 요약

원본 4만 명 `.psv`를 **빈칸 그대로 한 번 캐싱**해두고, 주문(분할·모델)마다 그 캐시에서 꺼내 다듬는다. 피처는 활력+나이·성별 기본에 핵심 검사 6종을 옵션으로, 결측은 모델별로 다르게, 분할은 환자 단위 + 병원 인식으로 3가지 모드를 연다.

## H1 범위 / 범위 외

| 범위 (H1이 하는 것) | 범위 외 (다른 H) |
|---|---|
| 피처 선택·피처셋 분기 셋업 | 실제 학습 (H2) |
| 결측 처리(모델별 분기) | ablation **실행** (H2) — 분기 셋업만 H1 |
| 정규화·클리핑 | utility·cross-site **평가 실행** (H3) — 분할 모드만 H1 |
| 윈도잉 | 마스크 누수검증 (H3) — 자리만 H1 |
| 분할(모드 3종) | 서빙·운영 (H4) |
| 캐싱 레이어 | |

## 결정 1: 피처 선택 — 활력 8 + 나이·성별 2 + 핵심 검사 6 (최대 16), 피처셋 2분기

- **결정**: 후보 피처를 16개로 한정한다. 활력 8(HR, O2Sat, Temp, SBP, MAP, DBP, Resp, EtCO2) + 나이·성별 + 핵심 검사 6(WBC, Lactate, Creatinine, BUN, Platelets, Glucose). 피처셋을 config로 2분기: **활력셋(10)** vs **활력+검사셋(16)**. ICULOS·Unit1·Unit2·HospAdmTime은 제외.
- **근거 + 출처등급**:
  - 검사 대부분이 90%+ 결측이라, 다 쓰면 입력이 "추측한 값" 천지가 됨 [확인됨: EDA — 측정컬럼 평균 결측 80.1%, 검사 26종 대부분 >90%].
  - ICULOS 제외 = 우측 절단 컨닝 차단 [확인됨: research/04 + EDA §6 — 패혈증 환자 100%가 마지막 ≤10h 연속 양성, 우측 절단].
  - 핵심 검사 6종 선정 [우리 결정 — research/02].
- **고려한 대안**: (a) 검사 26종 전부 → 노이즈·추측 폭증으로 기각. (b) 활력만 고정 → 검사의 신호를 영영 못 봄. 그래서 "고정"이 아니라 **ablation으로 비교**(H2)하려고 2분기로 연다.
- **미결/옵션**:
  - **(A3) EtCO2 거취** — EDA상 **96.3% 결측 + 37,120명(전체 92%)이 한 번도 안 잼**. ffill로 끌어올 값조차 없어 거의 전부 train 평균으로 채워지는 *사실상 죽은 채널*. 활력 8에서 **뺄지(→활력 7)** 검토 필요.
  - **(옵션) ShockIndex 등 파생피처** — 결정 6 참고.
- **검토 요청 항목**:
  - 핵심 검사 6종이 **선행연구 피처 중요도와 일치**하는지 1차 출처로 교차 확인 (02·확인 과제).
  - EtCO2를 빼는 게 맞는지: 죽은 채널이 모델에 해가 되는지 vs 활력셋 정의 일관성.

## 결정 2: 결측 처리 — 모델별 분기, 0-fill 금지

- **결정**: XGBoost는 **NaN 그대로** 먹인다. GRU는 **환자 내 forward-fill**(과거→미래만) → 그래도 빈 칸은 **train split 평균**으로. **0으로는 절대 안 채움.**
- **근거 + 출처등급**:
  - XGBoost 결측 내장 처리 [ML 일반]. 미리 채우면 그 강점을 방해.
  - ffill이 챌린지 관행 [확인됨: research/02 — Pou-Prom[3]·Fu[4]].
  - GRU는 NaN 못 먹음 [ML 일반].
  - 0-fill 금지: 의료에서 0은 진짜 값(혈압 0 = 사망) [우리 결정 / 도메인 상식].
- **고려한 대안**: 두 모델에 동일 결측 처리 → XGBoost 강점 죽고 GRU와 한쪽이 불리. 통일이 오히려 틀림.
- **미결/옵션**: 없음.
- **검토 요청 항목**: ffill이 **미래→과거로 새지 않는지**(시간 정렬 방향), train 평균이 **train split에서만** 계산되는지(누수).

## 결정 3: 정규화 — train-only z-score + 가벼운 클리핑

- **결정**: 피처를 z-score 표준화. **평균·표준편차는 train split에서만** 계산. 생리적 범위 밖 극단값은 가볍게 클리핑.
- **근거 + 출처등급**:
  - train-only 정규화 = 누수 방지 [ML 기본 원칙].
  - 가벼운 클리핑이면 충분 [확인됨: EDA §7 — 범위 밖 값 0.1% 미만].
- **고려한 대안**: 전체 데이터로 정규화 → 시험 정보가 학습에 샘(누수). 공격적 이상치 제거 → EDA상 불필요.
- **미결/옵션**: 클리핑 경계 수치(예: HR 상한)는 구현 디테일 → 핸드오프에서 기준만.
- **검토 요청 항목**: cross-site 모드에서 정규화 통계가 **train 사이트에서만** 잡히는지(결정 5·8과 맞물림).

## 결정 4: 윈도잉 — WINDOW=8, stride=1, Method A(마지막 시각 라벨)

- **결정**: 길이 8시간·스텝 1의 **완전 윈도우**만 생성. 라벨 = 윈도우 **마지막 시각**의 SepsisLabel. 패딩 없음.
- **근거 + 출처등급**:
  - 8h면 환자 97.5%가 살아남고(≥12h) 실시간 조기경보에 충분히 짧음 [확인됨: EDA §4].
  - 모든 환자 ≥8h [확인됨: EDA — 최소 길이 8].
  - 스모크에서 배선 검증 완료 [확인됨: smoke/].
  - Method A(per-timestep) [우리 결정 — research].
- **고려한 대안**: 24~48h 윈도우 → 짧은 체류 환자 대량 손실(EDA). 첫 시각 라벨 → off-by-one 함정.
- **미결/옵션**: 없음.
- **검토 요청 항목**: 라벨 정렬이 **마지막 시각**이 맞는지(스모크 `dataset.py`와 일치), 우측 절단을 신호로 누수시키지 않는지.

## 결정 5: 분할 — 환자 단위 + 병원 인식, 모드 3종

- **결정**: 분할은 **환자(파일) 단위**. 모드를 3개 연다: **통합 random**(A·B 섞음) / **A→B**(A 학습, B 시험) / **B→A**. H1에서 셋 다 구현, H3는 호출만.
- **근거 + 출처등급**:
  - 환자 단위 = 누수 방지 [ML 기본].
  - 병원 인식 필요: A/B 패혈증 유병률 다름(A 8.8% vs B 5.71%) [확인됨: EDA §8].
  - cross-site가 7년째 미해결 핵심 [확인됨: research/01·04].
- **고려한 대안**: 행 단위 random → 같은 환자 양쪽 걸침(누수). 통합만 → 일반화 측정 불가.
- **미결/옵션**: val 비율, seed = 핸드오프 디테일.
- **검토 요청 항목**: 세 모드 모두에서 환자가 train/test에 **걸치지 않는지**, A→B에서 **B가 학습에 전혀 안 닿는지**.

## 결정 6: baseline 입력 표현 — 윈도우 요약통계 (XGBoost)

- **결정**: baseline은 **XGBoost**. 입력은 8시간 윈도우를 **요약통계**(마지막값·평균·최소·최대·기울기 등)로 압축한 한 줄. NaN은 그대로. 윈도우 경계(8h)는 **GRU와 동일**하게 맞춰 "같은 관측 구간"을 보장.
- **근거 + 출처등급**:
  - "윈도우를 요약해 트리에 먹이는 슬라이딩 윈도우 피처"는 챌린지 표준 레퍼토리 [확인됨: 2024 리뷰 — 참가팀들이 슬라이딩 윈도우 피처 + 부스팅 광범위 사용].
  - XGBoost 선택 [우리 결정 — 결측 내장 처리, 02 서술 기준].
  - ⚠️ **정직성**: 우승팀(Morrill)은 *단순 요약통계가 아니라* path signature + 도메인 파생 피처(ShockIndex·BUN/CR·PartialSOFA)를 씀 [확인됨]. 우리 baseline은 그보다 **가벼운 요약통계** 버전이며, "우승팀이 단순 요약통계를 썼다"는 주장은 하지 않는다.
- **고려한 대안**: (가) 시각별 행 → 추세 못 봄. (나) 윈도우 평탄화 128열 → 차원 폭발·순서 의미 상실·빈칸 8배. (다) 요약통계 채택.
- **미결/옵션**: **(옵션) ShockIndex(HR÷SBP) 등 도메인 파생 피처**를 요약통계에 얹을지. 우승팀이 검증했고 만들기 싸지만, baseline은 가볍게 가는 게 원칙이라 필수 아님.
- **검토 요청 항목**:
  - 슬라이딩 윈도우 피처가 챌린지 표준이라는 근거를 **1차 출처로 재확인**(리뷰가 아니라 실제 상위팀 코드/논문에서).
  - ShockIndex 등 파생피처를 baseline에 넣는 게 "가벼운 하한선" 원칙과 충돌하는지.
  - ⚠️ **벤치마크 함정**: AUC 0.98 류의 보고치는 누수 의심 → 목표로 삼지 말 것. 공식 utility(우승 ~0.36) 기준 유지.

## 결정 7: 결측 마스크 — 기본 OFF (옵트인 자리만)

- **결정**: 결측 마스크 채널은 **기본 미사용**. H1에서는 마스크 생성 코드 + off 플래그 **자리만** 만든다. 켜는 검증(검증 A·B)은 H3.
- **근거 + 출처등급**:
  - 마스크는 "의사가 언제 의심했나"를 베껴 **치료행동 누수** 통로가 될 수 있음 [확인됨: research/04 — Epic 외부검증 실패, Wong 2021].
  - 효과는 과제·환자군 의존 [확인됨: research/02 — GRU-D[5] vs Singh[6]].
- **고려한 대안**: 기본 ON → 누수 위험 미검증 상태로 채택. 아예 미구현 → H3 검증을 못 돌림.
- **미결/옵션**: 채택 여부 자체가 H3 검증 결과에 달림.
- **검토 요청 항목**: off가 진짜 기본인지(실수로 켜지지 않는지), 마스크 생성이 미래 정보를 안 쓰는지.

## 결정 8: 캐싱 레이어 — 16피처 NaN 보존 raw를 한 번만

- **결정**: 16개 후보 피처를 **NaN 그대로** + 라벨 + 사이트로, 환자별 raw 배열로 **한 번** 만들어 캐싱. 4만 파일 읽기는 이때 한 번뿐. 피처셋 선택·분할·결측처리·정규화·윈도잉은 모두 **런타임**(캐시에서 꺼내).
- **근거 + 출처등급**:
  - NaN 보존해야 XGBoost(그대로)·GRU(ffill) 두 경로를 한 캐시로 커버 [우리 결정].
  - 16개 풀셋 캐싱하면 활력셋(10)은 컬럼 슬라이스로 파생 [우리 결정].
  - 재학습 다회(ablation×cross-site) 비용 절감의 핵심 레버 [우리 결정 — GPU 불필요, CPU로 충분].
- **고려한 대안**: ffill·정규화까지 캐싱 → 모델별 분기·분할별 정규화를 못 함(누수/경직). 매번 파일 읽기 → 느림.
- **미결/옵션**: 캐시 포맷(npz/parquet) = 핸드오프 디테일.
- **검토 요청 항목**: 캐싱 경계가 **정규화를 분할에 제대로 묶는지**(train-only가 깨지지 않는지), 캐시에 **라벨·미래 정보가 새지 않는지**.
