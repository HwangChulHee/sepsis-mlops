# 개발 워크플로우 — 설계 결정 → 검토 게이트 → 구현

> 이 문서는 `sepsis-mlops`의 **풀 학습 단계(H1~H3)를 어떻게 개발하는가**에 대한 규칙이다.
> 한 번 정해두고 모든 H에서 재사용한다. 요약본은 루트 [`../../CLAUDE.md`](../../CLAUDE.md)에 있고, Claude Code는 매 세션 그걸 자동으로 읽는다.

---

## 0. 왜 이 워크플로우인가

설계 결정을 곧장 코드로 옮기면, **틀린 전제가 검증 없이 구현까지 흘러간다.** 실제로 이 프로젝트에서 "상위팀 관행 = 단순 요약통계"라는 설계 주장이 나왔다가, 1차 출처를 대조하니 *부분적으로 틀렸음*이 드러난 적이 있다(우승팀은 단순 요약통계가 아니라 path signature + 도메인 파생 피처를 썼다). 그 교정이 **구현 전에** 일어났기에 비용이 0이었다.

이 워크플로우는 그 교정 지점을 **모든 결정마다 강제로** 만든다.

---

## 1. 세 단계

```
[1] 설계 결정 문서(DDD) 작성   ← 사람 ↔ 채팅 Claude 핑퐁
        │
        ▼
[2] Claude Code 레드팀 검토     ← 게이트. 통과 못 하면 [1]로 복귀
        │  (1차 출처 + 레포 파일 직접 대조)
        ▼
[3] 구현 핸드오프 작성 & 실행   ← Claude Code 기계적 구현
```

- **[1] DDD**: "무엇을 왜 정했나"를 근거·출처등급과 함께 설명하는 문서. 검토 대상.
- **[2] 검토**: Claude Code가 **구멍을 찾는** 단계. 동의가 아니라 반박이 목적.
- **[3] 핸드오프**: 검토를 통과한 결정만 자립형 실행 문서로 묶어 구현.

> **두 종류 문서를 섞지 않는다.** *DDD = 설계를 정당화하는 문서*, *핸드오프 = 기계적으로 실행할 문서*. 성격이 다르므로 파일을 분리한다.

---

## 2. 역할 분리

| 역할 | 담당 | 하는 일 |
|---|---|---|
| **설계 핑퐁** | 사람 ↔ 채팅 Claude | 방향 정하기, 대안 비교, 결정 |
| **검토 게이트** | Claude Code (레드팀) | 1차 출처·EDA 대조, 구멍 찾기 |
| **구현** | Claude Code | 자립형 핸드오프 실행 |

채팅 Claude는 pdm-mlops 등 외부 레포를 볼 수 없고, Claude Code는 이 레포 파일과 1차 출처(웹)를 직접 볼 수 있다. 검토를 Claude Code에 맡기는 이유가 이것이다.

---

## 3. 출처 등급 (research/와 공통)

모든 결정의 근거에 등급을 단다.

| 표기 | 뜻 |
|---|---|
| **[확인됨]** | 공식 논문·챌린지 페이지·공식 코드를 직접 확인 |
| **[유도]** | 공식 파라미터로부터 계산(공식 코드로 재확인 권장) |
| **[우리 결정]** | 근거 위에서 우리가 내린 판단(논문 인용 아님) |
| **[검증 필요]** | 1차 출처로 재확인 예정/필요 |

검토 단계의 1차 목표는 **`[확인됨]`이 진짜 확인됐는지, `[검증 필요]`가 해소됐는지**를 가리는 것이다.

---

## 4. DDD 골격 — `h{N}_decisions.md`

각 결정은 아래 다섯 항목으로 적는다.

```markdown
### 결정 N: <한 줄 요약>
- **결정**: 무엇을 정했나
- **근거 + 출처등급**: 왜. [확인됨: 출처] / [우리 결정] / [검증 필요]
- **고려한 대안**: 무엇을 왜 버렸나 (검토자가 판단하려면 대안이 보여야 함)
- **미결/옵션**: 아직 안 정했거나 옵트인으로 둔 것
- **검토 요청 항목**: Claude Code가 특히 따져봐야 할 점
```

문서 맨 위에는 한 줄 요약과 "이 H의 범위/범위 외"를 적는다.

### DDD 개정 정책
검토 반려(HOLD) 후 수정은 **같은 파일에 덮어쓴다**(버전 파일 분리 금지 — `_v1`, `_v2` 만들지 않음). 문서 상단에 **개정 로그**(무엇을 왜 바꿨나)를 남기고, 이전 버전은 git history와 `h{N}_review.md`로 추적한다. 이렇게 해야 "지금 유효한 DDD가 무엇인가"가 항상 한 파일로 명확하고, 핸드오프 자립성이 깨지지 않는다.

---

## 5. 검토 프로토콜 — `h{N}_review.md` ⚠️ 핵심

**Claude Code에게 검토를 시킬 때 반드시 레드팀 모드로 지시한다.** 그냥 "검토해줘"라고 하면 동의만 하는 아첨 모드로 빠져 게이트가 무의미해진다.

### 검토 지시문(템플릿)

> 이 설계 문서(`design/h{N}_decisions.md`)의 **구멍을 찾아라. 동의가 목적이 아니라 반박이 목적이다.** 각 결정에 대해:
> - **(a) 출처 검증**: `[확인됨]` 항목은 1차 출처(논문/공식 페이지/공식 코드)를 직접 대조해 진위·정확성을 확인. 출처가 주장과 다르면 지적.
> - **(b) 논리 검증**: `[우리 결정]` 항목에 논리 비약·미검증 전제·반례가 없는지.
> - **(c) 누수 검증**: 이 결정이 새로운 데이터 누수 경로를 만들지 않는지(특히 환자 분할·정규화·피처·라벨 정렬).
> - **(d) 누락 검증**: 이 H 범위에서 빠진 결정이 없는지.
> - **(e) 데이터 정합**: `reports/eda_findings.md`의 우리 측정치와 모순되지 않는지.
> 각 지적은 **출처/파일 줄번호와 함께** 근거를 대라. 통과 항목과 보류 항목을 명확히 갈라라.

### review.md 산출 형식

```markdown
## 검토 결과 — H{N}
- 검토일 / 대상 commit
### 통과 (PASS)
- 결정 N: 근거 확인됨 — <어떻게 확인했나>
### 보류 (HOLD) — 설계 복귀 필요
- 결정 M: <문제> / <근거 출처> / <제안>
### 새로 발견한 누락·위험
- ...
```

**보류가 하나라도 있으면 [1]로 복귀**한다. 전부 PASS여야 [3] 핸드오프로 넘어간다.

---

## 6. 핸드오프 원칙 — `h{N}_handoff.md`

검토를 통과한 결정만 구현 핸드오프로 묶는다. 핸드오프는 **자립형**이어야 한다(Claude Code가 pdm-mlops 등 외부 레포를 못 보기 때문).

- 외부 레포 **참조 금지** — 필요한 구조·코드는 핸드오프 안에 직접 적는다(복붙).
- **실패 모드·성공 기준·범위 외**를 명시한다(프로그래매틱 PASS 기준 포함).
- 끝에 **`commit & push`**를 명시한다.
- 스모크가 검증한 배선(`smoke/`)은 재사용하되, 풀 학습 코드는 `src/sepsis/`에 새로 둔다.

---

## 7. 파일 네이밍 & 디렉토리

**스테이지별 폴더**로 묶는다 — 한 H의 라이프사이클 문서(decisions → review → handoff → handoff_review)가 한 폴더에 모인다. 폴더가 단계 맥락을 주므로 파일명에 `hN_` 접두는 붙이지 않는다.

```
design/
├── WORKFLOW.md          # 이 문서 (규칙, 한 번 만들고 재사용)
├── README.md            # 인덱스 맵 (스테이지·상태·결정 한눈에)
├── h1/
│   ├── decisions.md     # 설계결정문서(DDD)
│   ├── review.md        # DDD 레드팀 검토 결과
│   ├── handoff.md       # 구현 핸드오프
│   └── handoff_review.md # 핸드오프 레드팀 검토 결과
├── h2/  …  (동일 4종)
├── h3/  …
└── h4/                  # H4는 묶음 → 하위 스테이지로 중첩
    ├── serving/  (decisions·review·handoff·handoff_review)
    ├── drift/    …
    └── retrain/  …
```

- 각 스테이지 폴더 안의 4종 문서: **decisions**(설계 정당화·단일 진실원), **review**(DDD 게이트), **handoff**(자립형 실행 명세), **handoff_review**(실행 명세 게이트).
- 문서 간 링크는 **상대 경로**: 같은 폴더는 `decisions.md`, 다른 스테이지는 `../h2/decisions.md`, 루트 규칙은 `../WORKFLOW.md`(H4 하위는 `../../WORKFLOW.md`).
- 새 스테이지/하위 스테이지를 추가하면 폴더 하나를 새로 만들고 4종을 채운다.

### 레포 전체에서의 위치

| 디렉토리 | 성격 | 누가 쓰나 |
|---|---|---|
| `research/` | 근거 탐구 (왜 이 방법이 맞나, Q1~Q5) | 사람 ↔ 채팅 |
| `design/` | 실행 결정 (DDD·검토·핸드오프) | 사람 ↔ 채팅 ↔ Claude Code |
| `reports/` | 측정·검증 결과 (EDA·스모크) | Claude Code |
| `smoke/` | 시제품 파이프라인 | (완료) |
| `src/sepsis/` | 풀 학습 코드 | Claude Code (핸드오프로) |

`research`는 재료, `design`은 거기서 뽑은 레시피다. 섞지 않는다.

---

## 8. H 단계 개요 (범위)

풀 학습 = H1~H3(모델 확정까지). 운영(H4)은 별도 묶음 — 같은 3단계 프로세스를 서빙·드리프트·재학습 하위 스테이지에 재사용해 **구현 완료**(MLOps 루프 폐쇄).

| H | 이름 | 끝에서 사람에게 돌아오는 결정 | 상태 |
|---|---|---|---|
| **H1** | 전처리·피처·분할 | (분할/피처 셋업 확인) | ✅ 완료 |
| **H2** | 학습 (baseline + GRU + ablation) | **어느 피처셋을 메인으로?**(미결) | ✅ 완료 |
| **H3** | 평가 (PR-AUC·utility·cross-site·누수검증) | **마스크를 켤까?**(OFF) | ✅ 완료 |
| **H4** | 운영 (서빙·드리프트·재학습) | (루프 폐쇄) | ✅ 완료 |
