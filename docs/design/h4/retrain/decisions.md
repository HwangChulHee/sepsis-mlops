> **설계 근거**: H4-서빙·드리프트 위. drift watch 신호(`drift/watch.py`)를 받아 조사·재학습 결정을 사람에게 차려주고, 트리거되면 재학습→안전 교체. **MLOps 루프를 닫는 H4 마지막 단계.**
> **워크플로우·출처등급**: [`WORKFLOW.md`](../../WORKFLOW.md). 검토(`docs/design/h4/retrain/review.md`) 통과 후 핸드오프로.
> **상태**: 초안 — 레드팀 검토 전.
> **개정 이력**
> - **v2 (2026-06-28)** — review `8589aa3`의 HOLD 2건 반영
>   - HOLD 1(안전 교체·롤백): export가 고정 dir(gru_vitals) 덮어쓰기라 롤백 불가 → **버전드 번들**(`gru_vitals@<ts>`), RUN 버전 지정 원자 스왑, 이전 보존. PASS에 "이전 보존+살아있는 번들 미덮어씀".

> - v1: 초안.
> **H4 분할**: 서빙(완료) → 드리프트(완료) → 재학습(본 문서). 루프 폐쇄.

## 한 줄 요약

drift watch를 **재학습 행동으로 잇되, 결정은 사람(모델러)이 한다**(human-in-the-loop). 시스템은 **드리프트를 주 신호**(입력 분포 변화 = 성능 하락 *위험* 신호, ≠ 성능 하락 확정)로 "조사 권고(action)"를 올리고, 지연 라벨 백필로 성능을 **보조** 측정한다. 재학습 파이프라인·안전한 번들 교체(검증·롤백)를 갖추되, **자동 교체는 하지 않는다**(의료 — 새 모델이 더 나쁠 수 있고, 원인 조사·검증은 사람 몫). 우산장수(피드백 루프) 편향은 **인지·명시**하되 데이터(개입 컬럼) 부재로 태깅은 미구현. 우리 셋엔 신규 운영 데이터가 없어(A/B 고정) **B를 운영 데이터로 가정**(retrain/holdout 분할)해 재학습을 시뮬레이션 — **봉인 B는 검증 반복 소모가 아니라 학습/홀드아웃 재료로 역할 전환**. 교체는 **버전드 번들**로 이전 보존·롤백.

## 범위 / 범위 외

| 범위 (H4-재학습) | 범위 외 |
|---|---|
| watch→action 승격(드리프트 종합 → 조사 권고) | 완전 자동 재학습 루프(의료 — 금지) |
| 지연 라벨 백필 구조(성능 보조, 시뮬레이션) | 우산장수 완전 태깅(개입 데이터 없음 — 한계 명시) |
| 재학습 파이프라인(사람 트리거 시 실행) | concept drift 본격 대응 |
| 안전한 번들 교체(검증·롤백, export 패턴) | 실제 병원 라벨·개입 파이프라인 |
| human-in-the-loop 체크포인트 | 피처/결측 ablation 2차 바퀴 |

## 결정 1: 트리거 철학 — 드리프트 주도, 성능 보조 (★드리프트 ≠ 성능 하락)

- **결정**: 재학습 검토의 **주 신호 = 드리프트**(watch→action). 드리프트는 "성능이 낮다"가 아니라 **"성능이 떨어질 위험이 높아졌다"**는 조기 경보(연기 감지기). 성능(지연 라벨)은 **보조** — 늦게 오고 우산장수로 편향돼 단독 신뢰 불가. action은 "**무조건 재학습**"이 아니라 "**사람이 조사하라**"는 권고.
- **근거 + 출처등급**:
  - 드리프트 ≠ 성능 하락(입력 변해도 강건할 수 있음) [확인됨: 1차 출처 — statstest data≠concept]. 단 분포가 학습과 멀어지면 성능 하락 위험은 우리 데이터로 입증(H3 A→B 0.40→0.25) [확인됨: docs/reports/h3_results.md].
  - 성능 신호는 라벨 지연·우산장수로 못 믿을 이유 많음 → 보조 [확인됨: 1차 출처 + 우리 결정].
  - 드리프트 단독 ≠ 알람(거짓경보) → action도 "조사 권고"지 자동 재학습 아님 [확인됨: 1차 출처 futureagi].
- **고려한 대안**: 성능 주도 트리거(라벨 지연·우산장수로 늦고 편향 — 기각). 드리프트→자동 재학습(원인 조사 없이 버그·센서고장도 학습 — 기각).
- **검토 요청 항목**: watch→action 승격 조건이 "조사 권고"에 머무는지(자동 재학습으로 월권 안 하나).

## 결정 2: human-in-the-loop — 시스템은 정보·파이프라인, 결정은 사람 (★의료)

- **결정**: 재학습 **결정은 모델러(사람)**가 한다. 시스템은 (a) 신호·근거 제시(드리프트 종합·성능 보조·드리프트 원인 후보), (b) 사람이 트리거하면 재학습 실행, (c) 안전한 교체. **자동 모델 교체 금지.** 사람 체크포인트: 조사 판단 / 재학습 트리거 / 교체 승인.
- **근거 + 출처등급**:
  - 의료라 자동 교체 위험 — 새 모델이 더 나쁠 수 있고, 책임 소재·원인 조사(진짜 변화 vs 센서고장 vs 파이프라인 버그)는 사람 몫 [우리 결정 — 의료 도메인 / Epic 교훈].
  - 워크플로우의 자동진행 vs 사람 체크포인트 구분 연장 [확인됨: WORKFLOW.md].
- **고려한 대안**: 완전 자동 루프(human-out-of-the-loop) — 의료 부적합, 버그 학습 위험. 기각.
- **검토 요청 항목**: 사람 체크포인트가 명확히 강제되나(자동으로 넘어가는 경로 없나).

## 결정 3: 지연 라벨 백필 — 성능 보조 측정 + 우산장수 인지(태깅 미구현)

- **결정**: 라벨이 늦게 오므로(실시간 모름, 사후 확정) **지연 라벨을 백필해 사후 성능 측정**(보조). **우산장수 편향**(모델 적중→의사 개입→라벨 음성→성공이 FP로 집계)은 **인지·문서화하되 태깅 미구현** — PhysioNet에 개입/처방 컬럼이 없어 태깅 근거 자체가 없음. 대신 **성능을 주 트리거로 쓰지 않음**(결정 1)으로 편향 영향 최소화. 한계 명시.
- **근거 + 출처등급**:
  - 라벨 지연 → 백필 구조 [확인됨: 1차 출처 runbook]. 우산장수 = 피드백 루프 라벨 변질 [우리 결정 — 메모리 원칙].
  - 개입 데이터 부재 → 완전 태깅 불가, 한계 명시가 정직 [확인됨: PhysioNet 컬럼 / 드리프트 검토 결론].
  - 사람이 결정(결정 2)하니 시스템이 완벽 보정 불필요 — 편향 "인지"를 사람에게 전달하면 충분 [우리 결정].
- **고려한 대안**: intervention-confounded 완전 태깅(개입 데이터 없어 근거 없음 — YAGNI, 기각). 성능 무시(드리프트만 — 보조 가치 상실). 우산장수 무시(편향 모른 채 성능 신뢰 — 위험).
- **검토 요청 항목**: 태깅 미구현이 한계로 명시되나, 성능이 보조에 머무는지(주 트리거 아님).

## 결정 4: 재학습 파이프라인 — H1~H2 학습 재사용 + 새 데이터

## 결정 4: 재학습 파이프라인 — H1~H2 재사용 + B를 운영 데이터로 가정

- **결정**: 사람이 트리거하면 **H1~H2 학습 파이프라인 재사용**으로 재학습(전처리·학습·평가 동일, **배포 중 조합만** 재실행 — 전체 6조합 아님). **★ 데이터: 우리 셋엔 "신규 운영 데이터"가 없음(A/B 고정)** → **B를 운영 데이터로 가정**(A로 학습한 모델이 운영 중 마주친 다른 분포 — H3에서 cross-site로 입증). **B를 retrain/holdout으로 분할** → **A-train + B-retrain으로 재학습**, **B-holdout으로 검증**. 누수 규칙: train-only 전처리 통계 재산출, **B-holdout은 B-retrain과 분리된 새 홀드아웃**(검증 전용), 재학습 후 reference 갱신.
- **★ B 역할 전환 명시**: H3에서 B = cross-site 평가용 봉인(1회 관찰, 모델 선택엔 미사용) → **H4-재학습에서 B = 운영 데이터**(재학습 입력 + 자체 holdout 검증). 봉인 B를 매 재학습 검증에 **반복 소모하던 누수**(selection-on-test) 해소 — B는 이제 검증 대상이 아니라 학습/홀드아웃 재료.
- **근거 + 출처등급**: H1~H2 재현 가능 [확인됨: 코드]. B가 A와 다른 분포 = 운영 신규 데이터의 성질 [확인됨: docs/reports/h3_results.md cross-site]. H3에서 B로 모델 선택 안 하고 관찰만 했으므로 재학습 입력 전환 가능 [확인됨: h3_results.md]. train-only는 누수 원칙 [확인됨: H1·H3].
- **고려한 대안**: 봉인 B를 매번 검증에 재사용(selection-on-test 누수 — HOLD 2, 기각). 신규 운영 데이터 홀드아웃(실재 안 함 — A/B 고정, 기각). setA 내부 분할 시뮬(B의 다른 분포 성질 못 살림).
- **미결/옵션**: B의 retrain/holdout 분할 비율·방식(환자 단위)은 핸드오프. 실운영 시엔 진짜 신규 데이터가 이 역할(우리는 B로 시뮬레이션).
- **검토 요청 항목**: B-retrain/holdout 분할이 환자 단위로 누수 없나, B 역할 전환이 H3 결과와 모순 없나.

## 결정 5: 안전한 교체 — 검증 후 export 번들, 롤백

- **결정**: 재학습 모델은 **검증 통과 후에만** 교체. 교체 = export 번들 패턴(`scripts/h4/h4s_export_bundle.py`)을 **버전드로 확장** — 고정 dir 덮어쓰기가 아니라 **`gru_vitals@<timestamp>` 버전 dir**에 새 번들 생성(이전 버전 보존, 살아있는 번들 미덮어씀), ConfigMap RUN을 **버전 지정**으로 전환(원자 스왑). **검증 게이트**: 새 모델이 기존보다 (A-val + **B-holdout**)에서 나쁘지 않을 것(수치 임계 핸드오프). **롤백**: RUN을 이전 버전 dir로 되돌림(이전 번들 보존돼 가능). 교체 승인 = 사람.
- **근거 + 출처등급**: 현재 export가 고정 dir rmtree+재생성(덮어쓰기, 버전 없음) → 롤백 불가 [확인됨: export 코드]. 버전드 dir + RUN 버전 전환이라야 이전 보존·원자 스왑·롤백 [확인됨: H4s-c ConfigMap RUN=dir 이름]. 검증 없는 교체는 더 나쁜 모델 배포 위험 [확인됨: 1차 출처 canary/rollback].
- **고려한 대안**: 고정 dir 덮어쓰기(롤백 불가·살아있는 번들 파괴 — HOLD 1, 기각). 검증 없이 교체(위험). 롤백 없음(복구 불가).
- **미결/옵션**: shadow/canary는 핸드오프. 검증 수치 임계. 버전 보존 개수.
- **검토 요청 항목**: 버전드 번들이 이전 보존+살아있는 번들 미덮어씀을 보장하나, 롤백이 RUN 전환으로 실제 가능한가, 검증이 B-holdout까지 보나.

## 부속 결정

- **MLOps 루프 폐쇄**: 서빙(예측) → 드리프트(감시) → 재학습(행동) → 새 번들 → 서빙. 본 단계가 루프를 닫음.
- **우리 데이터 한계 정직**: 실제 라벨·개입 스트림 없음 → 지연 라벨·개입을 **시뮬레이션**해 구조 검증. "갖추되 한계 인지".
- **드리프트 reference 갱신**: 재학습 후 새 A-train 분포로 drift reference 갱신(옛 기준 잔존 방지).

## H4-재학습 PASS 기준 (핸드오프에 박을 게이트 — 초안)

1. watch→action 승격이 "조사 권고"에 머묾(자동 재학습 아님). 사람 트리거 필요.
2. 지연 라벨 백필 구조(시뮬레이션) — 성능 보조 측정, 우산장수 태깅 미구현이 한계로 명시.
3. 재학습 파이프라인이 H1~H2 재사용(배포 조합만) + train-only 누수 규칙 준수. **B를 운영 데이터로 가정, B-retrain/holdout 환자 단위 분할**(holdout은 검증 전용, retrain과 분리).
4. **안전 교체**: **버전드 번들**(`gru_vitals@<ts>`, 이전 보존+살아있는 번들 미덮어씀), 검증 게이트(기존보다 A-val+**B-holdout** 나쁘지 않음) 통과 후에만 교체, **롤백(RUN 이전 버전 전환) 가능**, 승인=사람.
5. 재학습 후 drift reference 갱신(서빙 번들과 정합).
6. **human-in-the-loop**: 조사·트리거·교체 승인이 사람 체크포인트(자동 재학습·교체 경로 0건, grep 강제).
