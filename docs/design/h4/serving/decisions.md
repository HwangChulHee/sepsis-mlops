## 한 줄 요약

H2 GRU를 **실시간 서빙**한다 — 환자 시점 데이터가 스트리밍으로 도착하면 매시점 패혈증 위험확률 + τ 기반 알람을 응답. **A 동결 전처리·τ를 그대로**(train-serving skew·누수 방지), **단방향 causal**(미래 안 봄), 마스크 OFF(H3 확정). 스트리밍 시뮬레이터로 시간 흐름을 재현하고, 관측성 훅(Prometheus)을 심어 다음 단계(드리프트·재학습)의 토대를 만든다. **운영 본체 — 모델 성능이 아니라 운영 설계가 차별점**(Epic 실패 vs TREWS 교훈).

## 결정 1: 서빙 대상 모델 — GRU(기본 vitals), 설정으로 교체 가능

- **결정**: 서빙 모델 = **GRU**(B cross-site 최고). 기본 run = **gru/vitals**(B에서 vitals_labs보다 전이 우수). **설정은 단일 run을 선택**하고, 그 run에서 **model+featureset+전처리통계(μ/σ·fill·clip)+τ+input_dim을 원자적으로(통째) 로드** — model·featureset을 독립 노브로 두지 않는다(독립이면 불일치 번들 = train-serving skew). 교체는 `run=gru_vitals ↔ gru_vitals_labs`처럼 run 단위. H2 아티팩트가 이미 run 단위 번들이라 자연스러움.
- **근거 + 출처등급**:
  - GRU가 A-val·B 모두 최고(B util 0.247 vs 트리 0.055) [확인됨: reports/h3_results.md].
  - vitals가 cross-site 전이 우수(gap 0.162 < vitals_labs 0.265) [확인됨: h3_results.md].
  - 피처셋 미결 → 교체 가능 설계가 "재설정 가능한 운영환경" 메시지 [우리 결정].
- **고려한 대안**: 모델 하드코딩(교체 불가, 미결 피처셋과 모순). 앙상블(복잡도↑, 범위 외).
- **검토 요청 항목**: run 선택이 model+featureset+전처리+τ+input_dim을 **동일 run에서 원자적으로** 로드하는지(불일치 번들 불가).

## 결정 2: 입출력 계약 — 매시점 위험확률 + τ 알람

- **결정**: 입력 = 환자 한 명의 **현재 시점 활력(+옵션 검사) 한 줄**(+ 환자 식별자로 누적 상태 관리). 출력 = **그 시점 패혈증 위험확률 p(0~1)** + **알람(p ≥ τ)**. τ는 A-val 동결값. 매시점 호출(스트리밍).
- **근거 + 출처등급**: m2m은 매시점 예측 [확인됨: H1 결정 4]. τ는 A-val 동결(H3 규칙) [확인됨].
- **고려한 대안**: 전체 시퀀스 일괄 입력(배치 평가용이지 실시간 아님). 확률 없이 이진만(임상 활용도↓).
- **검토 요청 항목**: 알람 임계가 A-val τ로 고정인지(운영 데이터로 재선정 금지 — 누수).

## 결정 3: 스트리밍 상태 관리 — hidden state 이어가기 (causal)

- **결정**: 단방향 GRU라 시각 t 예측에 1..t만 필요. 스트리밍에서 **환자별 hidden state를 보존**해 새 시점 1스텝씩 전진(O(1)/시점). 매번 전체 시퀀스 재입력(O(t)) 대비 효율적. **두 방식이 수치 동일**해야(검증).
- **근거 + 출처등급**: 단방향 causal이라 hidden state 이어가기가 전체 재입력과 동치 [확인됨: GRU 구조 / H1 결정 4 단방향]. 실시간엔 O(1)이 필수 [우리 결정].
- **고려한 대안**: 매시점 전체 재입력(단순하나 긴 환자에서 느림). 윈도우 한정(초기맥락 손실).
- **미결/옵션**: 환자별 상태 저장소(메모리/Redis 등)는 핸드오프 디테일.
- **검토 요청 항목**: hidden-state 이어가기 == 전체 재입력 결과 동일(누수 없는 causal)인지 검증 명세.

## 결정 4: train-serving 일관성 — A 동결 전처리 재현 (★누수 방지)

- **결정**: 서빙 전처리 = 학습과 **동일 파이프라인·동일 상수**. **ffill(스트리밍에선 환자별 "직전 관측" 상태 유지) → fill(A-train 평균) → clip(A 범위) → z-score(A μ/σ)**. 전부 **A 동결값**, 운영 데이터로 재계산 금지. **마스크 OFF**(H3 확정). **0-fill 금지**.
- **근거 + 출처등급**:
  - train-serving skew = 운영 실패 주원인. 동일 전처리라야 학습 분포와 일치 [확인됨: ML 일반 / Epic 교훈].
  - A 동결값 사용 = H3 누수 규칙 연장(운영 데이터로 재정규화하면 누수) [확인됨: H3 결정 2].
  - 마스크 OFF = cross-site 악화로 기각됨 [확인됨: H3-c].
- **고려한 대안**: 운영 데이터로 정규화 재계산(분포 적응처럼 보이나 train-serving skew+누수). 기각.
- **검토 요청 항목**: 서빙 전처리가 학습과 bit-동일인지(특히 ffill 스트리밍 상태), 0-fill 부재.

## 결정 5: API 설계 — FastAPI

- **결정**: **FastAPI**. 엔드포인트: `POST /predict`(시점 데이터→위험확률+알람), `GET /health`. 환자 상태는 식별자로 관리. pydantic 입력 검증.
- **근거 + 출처등급**: FastAPI = 경량·async·pydantic 검증 [우리 결정 · pdm-mlops 경험]. 
- **고려한 대안**: Flask(async 약함), gRPC(범위 과함).
- **미결/옵션**: 동기/배치 엔드포인트 추가는 핸드오프.
- **검토 요청 항목**: 입력 스키마가 피처셋 교체와 호환되는지.

## 결정 6: 스트리밍 시뮬레이터 — 시간순 도착 재현

- **결정**: 환자 기록(예: B의 일부 또는 합성)을 **시간순으로 한 시점씩** API에 흘려보내는 시뮬레이터. 매시점 응답을 수집해 "실시간이라면 어땠을지" 재현. **미래 미사용**(causal 보장).
- **근거 + 출처등급**: 실시간 운영을 오프라인에서 검증하는 표준 방식 [우리 결정]. m2m·causal 정합 [확인됨].
- **고려한 대안**: 일괄 배치(실시간성 미검증).
- **미결/옵션**: 시뮬레이터 입력 소스(B vs 합성) — **B 사용 시 채점 아닌 "재생"만**(선택·튜닝에 미사용, H3 규칙).
- **검토 요청 항목**: 시뮬레이터가 미래를 흘리지 않는지, B 사용 시 관찰 전용인지.
