## 한 줄 요약

운영 입력이 학습 분포(A-train RAW)에서 **벗어나는지를 라벨 없이 실시간 감시**한다. 거리지표(Wasserstein/PSI/JS, 효과크기 내재)로 피처별 드리프트를 재고, **검정 단위는 환자당 1관측**(시점 자기상관 회피), 거짓경보는 **합성 주입으로 경험적 보정**. 단 **드리프트 단독으로 알람하지 않고**(거짓경보 → 온콜 소진), 지연 라벨 백필로 측정한 성능 하락과 **결합**해야 알람 — 단 우산장수 피드백 루프 변질은 **태깅으로 보정**. Evidently로 감지(거리기반), Grafana로 대시보드. **"성능 하락이 모델 탓이 아니라 데이터 변화 탓일 수 있다"를 성능이 떨어지기 전에 경보**해 재학습 타이밍을 잡는다.

- **근거 + 출처등급**: "모델이 본 세상"에서 벗어나나가 우리 질문 [우리 결정]. 학습 분포 대비가 표준 [확인됨: 업계 관행]. cross-site(A→B) 붕괴를 H3에서 봤으므로 입력이 A에서 벗어나면 그 붕괴 위험 신호 [확인됨: reports/h3_results.md]. **결측률 site-specific** [확인됨: reports/h3_results.md H3-c — 마스크 cross-site 악화].

- **검토 요청 항목**: raw reference가 serve 입력 raw_row와 같은 단계인지, 결측률 reference가 INPUT_MISSING 카운터와 연결되나.

## 결정 2: 현재 윈도우(current) — 최근 N, 의료라 느린 주기

- **결정**: 현재 분포 = **최근 윈도우**(최근 N건 또는 최근 기간). 실시간 수집 + **주기적 분석**. 의료는 저변동 도메인이라 분석 주기를 촘촘히 안 함(분/시간 단위 알람 대신 일/주 단위 분석 기본, 설정 가능).
- **근거 + 출처등급**: 단일 시점은 노이즈, 윈도우로 안정화 [확인됨: 업계 관행]. 분석 주기는 도메인 의존 — 의료 등 저변동은 느림 [확인됨: 1차 출처]. 짧은 윈도우=민감(노이즈)/긴 윈도우=둔감(지연) 트레이드오프 [확인됨].
- **고려한 대안**: 매 예측 즉시 비교(노이즈 폭증). 고정 대형 윈도우(느린 변화 못 잡음).
- **미결/옵션**: 윈도우 크기·분석 주기 구체값은 핸드오프(설정). reference 대비 표본 크기 균형.
- **검토 요청 항목**: 윈도우 표본 크기가 검정 신뢰성과 거짓경보(결정 3)에 미치는 영향.

## 결정 3: 검정 — 거리지표(Wasserstein/PSI/JS, 효과크기 내재) ★대표본 거짓경보 방지

- **결정**: 피처별 **PSI + KS** 병용. PSI 실용 임계 **< 0.1 안정 / 0.1~0.2 조사 / > 0.2 조치**. **KS는 p값 단독 금지 — 효과크기(KS 통계량) 하한과 함께** 판정. (우리 표본이 크면 KS가 사소한 차이도 "유의"로 잡아 거짓경보.)
## 결정 3: 검정 — 거리지표(Wasserstein/PSI/JS, 효과크기 내재) ★대표본 거짓경보 방지

- **결정**: 피처별 드리프트는 **거리지표**로 잰다 — 수치형은 **Wasserstein 거리**(Evidently가 >1000 obs에서 기본 사용)/**PSI**, 범주·결측률은 **JS divergence**. **KS p값 단독 사용 금지** — 우리 표본이 크면 사소한 차이도 "유의"로 잡아 거짓경보(over-power). 거리지표는 **효과크기가 내재**돼 표본 크기에 휘둘리지 않음. PSI 실용 임계 **< 0.1 안정 / 0.1~0.2 조사 / > 0.2 조치**(관행, 도메인 보정).
- **근거 + 출처등급**:
  - **대표본 KS over-power**: p<0.001인데 KS 통계량 0.02처럼 통계적 유의·실질 무의미 → KS 부적합 [확인됨: 1차 출처 statstest]. Evidently가 대표본에서 거리(Wasserstein)로 가는 이유가 이 함정 회피 [확인됨: Evidently 문서].
  - PSI 임계 0.1/0.2 [확인됨: 업계 관행]. 단 rule-of-thumb이라 도메인 보정 필요(arXiv 사례 1.4~5.2) [확인됨: 1차 출처].
- **고려한 대안**: KS p값+Bonferroni(자기상관·Evidently 거리기반과 충돌, over-power 재발 — 기각). PSI 단독(범주·결측엔 JS 병행).
- **검토 요청 항목**: 거리 임계가 합성 주입(결정 4·PASS #6)으로 경험적 보정되나, raw 윈도우(결정 6)로 계산되나.

## 결정 4: 거짓경보 통제 — 합성 주입 경험적 보정 + 검정 단위(자기상관)

- **결정**: 거리지표라 분석적 α(p값)가 없으므로, **거짓경보는 합성 드리프트 주입으로 경험적 임계 보정**(PASS #6). 알려진 분포 이동을 넣으면 감지, 안 넣으면 거짓경보율이 목표(≈5%) 근처가 되도록 거리 임계를 조정. **검정 단위 = 환자당 1관측(예: 환자별 요약/마지막값) 또는 블록 보정** — per-timestep raw는 같은 환자 연속 시점이 강하게 자기상관이라 독립표본 가정이 깨져 실제 FPR ≫ 목표. 시점 단위로 검정 금지.
- **근거 + 출처등급**:
  - 거리지표는 p값 없음 → 분석적 α·Bonferroni 무의미, 경험적 보정이 맞음 [확인됨: Evidently 거리기반]. metrics.py는 per-timestep raw 관측이라 자기상관 [확인됨: 코드].
  - 자기상관 표본을 iid로 검정하면 거짓양성 폭증 [확인됨: 통계 일반].
- **고려한 대안**: 시점 단위 검정 + Bonferroni(자기상관으로 FPR 폭증, 거리지표에 p값도 없음 — 기각). 분석적 α 고정(거리지표에 부적용).
- **검토 요청 항목**: 환자당 1관측 집계가 드리프트 신호를 죽이지 않나(요약 방식), 합성 주입 보정이 프로그래매틱한가.

## 결정 5: 알람 조건 — 드리프트 + 성능 결합 (★드리프트 단독 금지)

- **결정**: **드리프트 감지만으로 알람하지 않는다.** **입력 드리프트 + (지연 라벨 백필로 측정한) 성능 하락**이 결합될 때 알람. 라벨 없는 동안은 **드리프트를 "경보(watch)"로만**, 성능 영향 확인 시 "조치(action)"로 승격.
- **근거 + 출처등급**: 평가 영향 없는 드리프트는 거짓경보이고 온콜을 소진 [확인됨: 1차 출처 futureagi]. 라벨 없는 감시 = 프록시 + 사람 검토 샘플링 + 지연 라벨 백필 [확인됨: 1차 출처 runbook].
- **⚠️ 피드백 루프(우산 장수) 편향 보정**: 모델이 맞혀 의사가 개입→환자 회복→라벨이 "음성"으로 변질 → 모델 성공이 FP로 집계돼 성능이 체계적 하향 편향. **인지만으론 불충분 — 최소 태깅 필수**: 알람+개입이 일어난 케이스를 `intervention-confounded`로 표시해 윈도우 성능 집계에서 제외/별도 집계. 개입 데이터가 없으면(PhysioNet엔 처방 컬럼 없음) **한계 명시 + 개입/비개입 코호트 분리**. **편향 처리 없이 결합 알람 PASS 금지.** [우리 결정 — 메모리 원칙 + review]
- **고려한 대안**: 드리프트 단독 알람(거짓경보 천지, 온콜 소진). 성능만 감시(라벨 지연으로 늦음 — 드리프트의 조기경보 가치 상실). 우산장수 인지만(편향 둔 채 알람 — 기각).
- **검토 요청 항목**: 라벨 지연 구조에서 백필·watch→action 승격이 구현 가능한지, 우산장수 변질을 어떻게 표기하나.

## 결정 6: 데이터 소스 — raw 윈도우 보관 (Prometheus 히스토그램은 대시보드용)

- **결정**: 드리프트 검정은 **raw 입력 윈도우**(최근 N건 원값)로 계산. `serve/metrics.py`의 coarse Prometheus 히스토그램(버킷 [0,1,5,…300])은 **대시보드 시각화·운영 모니터링용**이고, 정밀 PSI/KS는 raw 윈도우로(coarse 버킷은 정밀도 부족). 윈도우 저장은 경량(최근 N만).
- **근거 + 출처등급**: Evidently 등은 reference vs current raw를 받아 검정 [확인됨: 도구 관행]. coarse 버킷으로 PSI 계산하면 버킷 경계 왜곡 [우리 판단]. metrics.py 주석도 "H4-drift can refine" 명시 [확인됨: serve/metrics.py].
- **고려한 대안**: Prometheus 히스토그램만으로 PSI(버킷 거칠어 부정확). raw 전량 보관(불필요, 윈도우면 충분).
- **검토 요청 항목**: raw 윈도우 저장이 서빙 상태(per-pid)와 분리되는지, 메모리/디스크 부담.

## 결정 7: 도구 — Evidently + Grafana

- **결정**: **Evidently**로 드리프트 감지 — 수치형 >1000 obs는 Wasserstein, 범주·결측은 JS, PSI 옵션(전부 거리지표, 효과크기 내재). reference vs current raw 윈도우 입력. 결과를 Prometheus로 노출 → **Grafana 대시보드**(피처별 거리·시계열·watch/action). 서빙 관측성 스택 재사용. **분석적 α/Bonferroni는 쓰지 않음**(거리지표라 p값 없음, 결정 4 경험적 보정).
- **근거 + 출처등급**: Evidently 기본이 거리지표(>1000 Wasserstein, ≤1000 KS, 범주 JS, 임계 0.1, α보정 내장 없음) [확인됨: Evidently 문서]. 직접 KS+Bonferroni 구현은 도구와 충돌·재구현 부담 → 거리기반 채택. Grafana는 pdm·서빙 사용 [우리 경험].
- **고려한 대안**: 직접 scipy로 KS/PSI(재구현 부담, Evidently가 표준). NannyML 등 대안(범위).
- **미결/옵션**: Evidently 버전·리포트 형식은 핸드오프(1차 확인).
- **검토 요청 항목**: Evidently 통합이 raw 윈도우(결정 6)·α 보정(결정 4)과 정합하나.

- **감시 단계**: covariate(입력)만 H4-드리프트 범위. concept drift(관계 변화)는 라벨 의존이라 H4-재학습/이연.

## H4-드리프트 PASS 기준 (핸드오프에 박을 게이트 — 초안)

1. A-train **RAW** reference 분포 + 결측률 산출·동결, 피처별.
2. 최근 윈도우 raw 수집 → **거리지표(Wasserstein/PSI/JS) 계산, 검정 단위=환자당 1관측/블록**(시점 자기상관 회피).
