## 한 줄 요약

Locust로 **"가상 사용자 = 환자 1명"**을 만들어 각 환자의 실제 setB PSV를 timestep 순서대로 `/predict`에 흘려보낸다(긴 스트림 지속). 동시 환자 수 N을 로그 스케일로 올려 **saturation knee**(RPS 정체 + p99 폭발)를 찾고, 코어 sweep으로 최적점을 본다. 측정 오염은 **Locust 내장 CPU 경고를 오라클**로 통제한다.

## 결정 2: 부하 형태 — 긴 스트림 지속 (환자=가상사용자)

- **결정**: Locust `User` 1명 = 환자 1명. 그 환자의 PSV를 timestep 순서로 `/predict`에 전송. **스트림 종료 시 반복하지 않고 아직 아무 User도 쓰지 않은 미사용 환자로 교체**(지속 부하는 교체로 유지). 동시 사용자 수 = 동시 환자 수 N. 램프업으로 점증.
- **★배타 배정 불변식 (M1)**: **각 Locust User는 실행 내내 배타적 distinct 환자를 점유**한다 — 두 User가 같은 pid를 동시에 밀면 per-pid 락(`predictor.py`의 `_lock(pid)`, `predict`가 `with self._lock(pid)`로 획득)이 직렬화는 하나 **두 스트림의 timestep이 뒤섞여 causal 붕괴**한다. setB 환자 20,000명 > N≤1000이므로 배정 가능([확인됨: `data/raw/training_setB/*.psv` 20,000개]). 교체 시에도 미사용 pid에서만 뽑아 배타성 유지.
- **★반복 재사용 금지 근거 (B1)**: 같은 pid를 다시 틀면 서버가 **이전 실행의 hidden state를 이어받아 곡선이 오염**된다 — 재사용 로더가 이 함정을 명문화한다 [확인됨: `psv_source.py:10-12` **F4** "서버엔 리셋 엔드포인트가 없어, 같은 patient_id 로 다시 틀면 서버가 이전 실행의 hidden state 를 이어받아 곡선이 오염된다"]. 실제 `app.py` 라우트는 `/predict·/health·/schema·/metrics·/admin/reload·/drift`뿐 — **`/reset` HTTP 엔드포인트 없음**([확인됨: `predictor.py:36 reset()`은 존재하나 라우트 미노출]). reset 엔드포인트 추가는 프로덕션 코드 수정 = (나) 범위 밖이므로, **미사용 환자 교체**로 회피한다. 교체는 클라이언트 **동시 점유 pid를 N개로 유계**로 묶어 같은 pid causal 오염을 막는다.
- **★서버측 pid 누적은 별개 — 유한 풀로 상한 (B1-r2)**: 교체가 **클라이언트 동시 점유만** 유계로 만들 뿐, **서버 dict는 옛 pid 엔트리를 GC 하지 않는다** [확인됨: `predictor.py:49 self._h[pid]=h_n`·`:33 self._locks[pid]=Lock()`·`preprocess_rt.py:43 self._last[pid]=state` — 모두 삭제 코드 없음, `reset()`은 `predictor.py:36`·`preprocess_rt.py:27`에 존재하나 HTTP 미노출]. 따라서 서버가 본 **distinct pid 총수는 동시 N이 아니라 누적** 증가한다. 단 미사용 환자 교체(엄격 "아직 아무 User도 안 쓴 pid에서만 뽑기")를 지키면 누적 distinct pid ≤ **setB 풀 20,000**으로 상한 → 서버 상태 메모리는 pid당 hidden(num_layers×hidden×4B ≈ ~1KB) + ffill 버퍼(F floats ≈ 수백 B) × 20,000 ≈ **수십 MB로 유상한**([유도], mem_limit 2g 대비 무시 수준). (drift window는 무한 누적 주체가 아님 — `window.py:26 deque(maxlen=5000)` 하드 캡·aux 게이트 `app.py:104` 뒤, m-r2-1.) 결정 6/7 해석 규약은 이 "누적이지만 유한 풀 상한" 사실 위에서 판정한다.
- **★지속시간·풀 고갈 정책 (B1-r2)** [우리 결정]: 위 상한은 "미사용 only"에서만 성립하고 **풀 20,000 소진 후 동작은 미정의**이므로 — (1) 각 부하 칸은 **풀 고갈 전 유한 지속시간**으로 돌린다(워밍업 컷 후 정상상태 창 확보하면 종료). (2) 지속·장시간 칸(SM-3 N=200 지속·무릎 탐색 연장)에서 풀 소진이 임박하면 **칸마다/코어 전환마다 서버 재시작**으로 서버 pid를 리셋한다(결정 5 N칸 스코프·코어 전환 재시작과 정합). (3) 고갈을 견디려 `run_suffix`(`psv_source.py:38`) 재활용으로 **무한 신규 pid**를 만드는 방식은 **택하지 않는다** — 택하면 서버 pid가 진짜 무한 증가해 결정 7 OOM이 되살아난다.
- **근거**: GRU는 환자별 hidden state를 이어가므로 한 환자 timestep은 **한 사용자가 순서대로** 밀어야 함(쪼개지면 causal 붕괴) [확인됨: `app.py:92-107` `/predict`가 `patient_id`별 `StatefulPredictor.predict`]. 병원은 환자들이 입원해 스트림을 이어가는 환경이라 버스트 아닌 지속이 현실 [우리 결정]. 순서 재생 로직은 `simulator.py:replay_patient` 참고 [확인됨]. 재사용 로더는 `replay/psv_source.PsvRowSource`(PSV→`{col: float|None}` 시간순 dict, `psv_source.py:41-48`) — `/predict` 스키마와 정확 일치 [확인됨].
- **요청 스키마**: `POST /predict` `{patient_id: str, features: {col: float|None}}` [확인됨: `app.py:77 PredictRequest`].

## 결정 4: ★측정 오염 방지 — Locust CPU 경고 오라클

- **결정**: 부하 중 측정 오염원을 다음으로 통제한다:
  1. **BLAS thrash** — `*_NUM_THREADS = cpus`(가에서 적용, 유지) [확인됨: SM-2].
  2. **CPU 제한** — 최상위 `cpus`(가에서 적용) [확인됨: SM-2].
  3. **★Locust↔serving CPU 쟁탈** — 같은 머신 POC로 두되, **Locust 내장 CPU 경고를 오라클**로 쓴다. 경고 없으면 부하 생성기가 병목 아님(오염 무시) → pinning 불필요; 경고 뜨면 코어 pinning 또는 별도 머신으로 격리 [확인됨: Locust 공식 문서 — CPU 소진 시 경고 로그, 무경고면 생성기 제한 아님]. **★무릎 탐색 국면 유의 (m2)**: 무릎 탐색은 수천 RPS라 생성기도 실경합 상태다 — Locust CPU 경고가 서빙 무릎보다 **먼저** 뜨면 그 무릎은 서빙이 아니라 **생성기의 무릎(무효)**이므로 폐기하고 격리 후 재측정한다. 경고 임계(기본 90% 미만)는 필요조건이지 충분조건이 아니다(sub-threshold 경고 사각 존재).
  4. **scrape 간섭** — 부하 중 per-patient gauge **off** + prometheus scrape **15s**(2s는 데모용) [우리 결정].
  5. **★워밍업 컷 정량 (M2)** — lazy-boot 오염을 다음으로 제거한다 [우리 결정]:
     - **(i) 프리웜**: 부하 시작 전 워밍 요청 1건으로 첫 `/predict`의 lazy-boot(300-trial 캘리브레이션 소진, `app.py:56-73` `state()`→`_load_all(force=False)`)를 미리 태워 첫 요청 수 초 지연을 부하 구간 밖으로 뺀다 [확인됨: `app.py` `state()`가 `"pred" not in _S`일 때 `_load_all` 호출].
     - **(ii) 컷 창**: 각 칸에서 램프업 완료 후 **첫 30초**를 측정에서 컷하고 정상상태만 집계.
     - **(iii) 통계 리셋**: 컷 창 경과 시점에 Locust 통계를 리셋(`--reset-stats` 또는 정상상태 창만 별도 집계)해 워밍업 표본이 백분위를 오염 안 시키게 한다.
     - 수치(1건 프리웜·30초 컷)는 [우리 결정] — SM에서 정상상태 진입이 더 늦으면 상향.
- **정량 근거**: 병원 부하 초당 <1 vs 단일 코어 Locust 천장 수천 RPS → 쟁탈 무시 수준, pinning은 사실상 YAGNI. 조기 격리 대신 오라클로 측정 후 판단(measure-then-scale) [우리 결정].
- **알려진 한계**: 같은 머신 = POC. 정석은 부하 생성기와 대상 분리(별도 머신) — 여건 시 로드맵 [우리 결정].

## 결정 5: 실험 매트릭스 — 십자(N축 × 코어축)

- **결정**:
  - **N축** (코어 2 고정): `1·10·50·200·500·1000` — **무릎 나올 때까지 연장**(안 나오면 상한 느슨하게 위로). N=1은 **최소 경합 기준선**(Locust User 1 + 서빙 1워커가 같은 호스트 코어를 공유하므로 경합 0이 아니라 최소 경합, m1).
  - **코어축** (N=200 고정): `1·2·4` — **스레드캡 동반 변경**(코어 바꿀 때 `*_NUM_THREADS`도 동수). 코어 전환마다 컨테이너 재시작 + 캘리브레이션 부팅 비용.
  - **★N칸 간 서버 상태 스코프 (M-r2-1)** [우리 결정]: N축 6칸은 코어 고정 시 컨테이너 재시작 없이 이어 도는데(재시작은 코어 전환에만 붙음), 그러면 **이전 칸의 서버 pid 상태(`_h`·`_locks`·`_last`)가 다음 칸으로 이월**된다 — 앞 칸 미사용 풀을 재할당하면 pid 재사용으로 옛 hidden state를 이어받아 causal 오염(B1-(A)) 재발, 전역 풀이면 6칸 누적이 20,000 풀 고갈을 앞당긴다. 따라서 **N칸마다(최소한 N을 크게 바꿀 때) 서버를 재시작**해 깨끗한 상태에서 측정한다 — 칸 간 pid 재사용·누적 이월을 차단하고, 코어 전환 재시작·B1-r2 고갈 정책과 하나의 재시작 규율로 정합시킨다.
  - 대표점 = 코어 2(실배포값·SM-2 확인) / N=200(무릎 근처 후보; 무릎이 더 낮으면 그리로 이동).
  - 총 ~8칸(십자, 코어2·N200 중복).
- **천장 정의**: 사용자 늘려도 RPS 정체 + p99 급증 = saturation [확인됨: Locust 공식 문서 — 무릎 판별 개념]. 무릎이 병원 부하의 몇 배인지가 여유의 크기.
- **★무릎 정량 임계 (M3)** [우리 결정]: 무릎은 헤드라인 산출물이므로 재현 가능한 기준으로 판정한다 — **N 2배 증가에 RPS 증가 <10%(정체)** *또는* **p99 > 최소경합 기준선(N=1)의 3배(급증)**를 만족하는 첫 N을 무릎으로 판정한다. 둘 중 먼저 걸리는 지점이 무릎. (수치 10%·3배는 [우리 결정]; Locust 문서는 정성 개념만 뒷받침.) **★기준선 표본수 (m-r2-2)**: N=1은 저 RPS라 컷 창 표본이 적으면 p99 추정이 흔들린다 — 기준선 p99는 **정상상태 창에서 최소 표본수를 충족한 구간에서 산정**한다(표본 부족 시 창을 늘리거나 p95로 대체). [우리 결정]
- **근거**: 풀 격자(18칸)는 코어 전환 비용 때문에 과함. 두 축 상호작용이 작다는 가설 하 대표점 sweep(YAGNI) — 상호작용 크면 격자 보충 [우리 결정].

## 결정 6: 측정 지표 — Locust 자동집계 + prometheus

- **결정**: 각 칸에서 **RPS · p50/p95/p99 · 에러율 · 메모리** 기록. Locust가 RPS·백분위·에러율 자동집계 [확인됨: Locust]. 메모리·CPU는 prometheus/`docker stats`. p99가 무릎의 선행 신호(큐 대기).
- **★메모리 해석 규약 (B1 → B1-r2 정정)**: R1은 "환자 교체로 pid 카디널리티 N개 유계 → 무한 누적 차단"으로 봤으나, 이는 **클라이언트 동시 점유(User=N)만 유계**로 만들 뿐 **서버 dict는 옛 pid를 GC 하지 않는다**(결정 2 "★서버측 pid 누적" 참조 — `_h`·`_locks`·`_last` 삭제 코드 없음, `reset()` HTTP 미노출 [확인됨]). 인과를 정정한다:
  - **서버 누적 distinct pid는 동시 N이 아니라 누적 증가**하되, 엄격 미사용 교체로 **유한 setB 풀(20,000)로 상한** → 서버 상태 메모리 ≈ 수십 MB로 유상한([유도], mem_limit 2g 무시 수준).
  - 따라서 지속 부하에서 서버 메모리가 **완만히 단조 상승하는 것은 정상**(누적 pid의 예상 아티팩트이지 시나리오 버그가 아니다). 동시 N 증가에 따른 증가분과 누적 pid 증가분이 **합산**된 값이 관측된다.
  - **진단 규칙은 반대다**: 완만한 단조 상승은 정상이고, **풀 상한(수십 MB)을 넘어 급격·무한 상승**해야 이상이다 — 무한 상승은 미사용 교체 불변식 위반(같은 User가 `run_suffix` 등으로 신규 pid를 무한 생성) 또는 풀 고갈 후 미정의 동작의 신호이므로 시나리오 버그로 취급한다.
  - (drift window는 여기서 무한 누적 주체가 **아니다** — `window.py:26 deque(maxlen=5000)` 하드 캡·aux 게이트 `app.py:104` 뒤라 상한이 있다 [확인됨, m-r2-1]. 무한 누적 가능 주체는 `_h`·`_locks`·`_last`뿐이고, 그마저 미사용 교체 하에선 풀 상한으로 유계다.)

## 결정 7: SM-3 닫기 — 부하 중 reload 2배 OOM

- **결정**: 대표 부하(N=200) 지속 중 `/admin/reload` 트리거 → 순간 메모리 증분이 `mem_limit`(2g) 안인지, OOM-kill 없는지 실측 [검증 필요]. (가)에서 부하 드라이버 부재로 이관한 항목을 여기서 종결.
- **★OOM 판정 = 누적분 분리 후 reload 순간 증분 (B1-r2)**: 관측 메모리에는 (a) **서버 pid 누적분**(풀 상한 수십 MB, 완만한 단조 상승, 결정 6)과 (b) **reload 순간 증분**(bundle 이중화 창, ~수백 MB 이하)이 섞인다. mem_limit 2g는 (a)+(b)를 모두 수용하나, **OOM 판정은 (a)를 baseline으로 빼고 "reload 순간 증분(b)이 OOM을 유발하지 않는가"로 규정**한다 — 누적분을 reload 탓으로 오귀속하지 않기 위함. (누적분이 풀 상한 수십 MB를 넘어 비정상 증가한 상태면 그건 결정 6의 시나리오 버그 신호이지 reload 판정 대상이 아니다.)
- **★합격 기준 = OOM/메모리, latency 아님 (m3)**: SM-3의 합격 판정은 **OOM-kill 무발생·mem_limit 내 수용**이다. reload 창 동안 캘리브레이션이 in-flight `/predict`와 CPU를 경합해 그 창의 p99가 팽창할 수 있으나, **핫패스(`/predict`)는 `_LOCK`을 획득하지 않으므로**(`app.py`의 `predict`는 `state()` 부팅 이후 `_LOCK` 미접촉; `_LOCK`은 `_load_all` 내부에서만) 락 스톨은 없다 [확인됨: `app.py:42 _LOCK`·`:55 with _LOCK`은 `_load_all` 전용, `predict:92-107`은 미획득]. → **reload 창의 latency는 결정 5 용량 매트릭스 p99에 합산하지 않는다**(별도 창으로 기록).
- **★"bundle 이중화 창" (m4)**: 핫스왑은 build-new-then-swap이라 옛/새 **번들**(가중치+reference+thr)이 순간 공존한다 [확인됨: `app.py:49-68 _load_all` — `new_s`/`new_ds` 빌드 후 `_S`/`_DS` 리바인드, 옛 dict는 리바인드 전까지 참조 유지]. 단 **torch 런타임(~248MB)은 프로세스에 1회 로드라 복제되지 않으므로** 실 peak는 "RSS 2배"보다 훨씬 작다(복제분은 bundle뿐) → 2g에 여유. "RSS 2배"가 아니라 **bundle 이중화 창**으로 해석. `mem_limit` 적용 자체는 SM-2 확인, 이중화 창 수용은 부하 하에서만 드러남.

## 알려진 한계 (정직)

- **단일 호스트** — network 성분 과소(minikube와 동일). 별도 머신 격리는 로드맵.
- **같은 머신 부하 생성기** — Locust CPU 경고로 오염 통제하나, 정석(분리) 아님. POC 한계.
- **GRU-only·워커1** — 이 스택의 실제 천장. 수평 확장(replica·Redis)은 로드맵. 측정은 "천장을 못 올리는 이유 = 상태"를 명시.
- **무릎 미도달 가능** — 부하가 너무 여유로우면 무릎이 N_max 위. 그 경우 "천장 > N_max, 병원 부하의 X배 여유"로 결론(여유 증명 성공).

## 레드팀 검토 요청

1. 시나리오에서 **한 환자를 한 사용자가 배타 점유**하는가(중복 배정 시 상태 오염)? PSV 순서 보장·환자 재사용(스트림 끝→반복) 방식이 상태를 오염 안 시키는가.
2. **Locust CPU 경고 오라클**이 실제로 오염을 잡기에 충분한가(경고 임계가 우리 부하에 유효한가).
3. **워밍업 컷·정상상태 정의**가 캘리브레이션·JIT 초기 오염을 제거하는가.
4. 코어 sweep 시 **스레드캡 동반 변경**이 매트릭스에 명시적으로 강제되는가(두 변수 혼입 방지).
5. SM-3 reload를 **부하 중** 트리거하는 방법이 측정을 왜곡 안 하는가.
